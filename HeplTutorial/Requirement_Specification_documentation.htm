1       Introduction

The purpose of this document is to collect, analyze, and define the high-level needs and features of CASE Tool, a visualized desktop application for for the creation of UCM's (use case maps). This document will describe the product functions, graphical user interfaces, requirements and constraints of the software. The details of how the CASE Tool fulfills these needs are detailed in the use case and more details will be available in the upcoming design phase. 

 

1.1    Purpose

This Software Requirements Document (SRD) describes the specification of the CASE Tool, which is in partial fulfillment of the requirements of COMP 354. It will define the high-level requirements of the user interfaces, product functions, user descriptions, assumptions and dependencies, constraints, specific requirements and an analysis model. The analysis model will include use case diagrams, class diagrams, sequence diagrams and state transition diagrams. Furthermore, a detailed project plan will be provided, including the schedule of the upcoming phases. This document is intended primarily for the members of Team 10 and the project coordinator, Dr. Juergen Rilling, as it will serve as a basis for the upcoming phases of the project. 

 

1.2    Scope

This document only addresses the high level requirements of the CASE Tool that will be used as a basis for the design phase. Screen shots of the user interfaces will give one an idea on how the game will look once it is completed. The product functions will include a prioritized list of which functions are to be included in the software. Specifications for the software will be outlined in the specific requirements section and the analysis model will contain UML diagrams. The use case diagrams will give an overview of the functions of the software and how the users will interact with the tools. The class diagrams will show the inter-relationships between the different objects in the software and the sequence diagrams will model the flow of logic within the software. 

 

The development plan will outline the three phases of the project with a Work Breakdown Structure (WBS) with a Gantt chart. It will include the amount of man-days to complete the project and is broken down into tasks. The numbers of resources required for the project are also listed in this section.

 

 

 

1.3    Definitions, Acronyms, and Abbreviations

Components

 

These are rectangular boxes representing data items such as components in a system performing particular functions or representing the systems themselves. Component boxes can be displayed within a general level component box containing these sub-boxes.

 

Responsibility Points

 

These indicate which components the use cases are operating on.

 

Scenario Paths

 

These are the lines representing the use cases. Each scenario path crosses over responsibilities and cannot exist without them.

2.      Problem description

2.1    Project Purpose, Scope, and Objectives

 

In order to create an efficient interface there needs to be a good graphical design, otherwise the user will have difficulty using it. This will result in lost productivity, user confusion and misunderstanding. To avoid such fundamental problems, Team TEN will design efficient and intuitive interfaces. This will include an intuitive interface that is uncluttered and easy to navigate and an action will be kept to the minimum. Team TEN will only use key or principle words for buttons and pop-up messages, which will be meaningful to the user. All input entered into input boxes will be fully validated by system. User will manually input through an input box, which will be validated by the system before being accepted. Some data will be entered by clicking on an icon or the use of a drop down list. To edit and draw map the user will also use mouse to select tools and drag, drop, resize, move objects. User will interact with the game through several interfaces, which will be detailed below. The four main

Interfaces are:

2.5.1.7          Use Case: Add Component

Description

Allows the user to add components to the Map.

Actors

Power User.

Pre-conditions

•User has opened the application.

•User has log in.

•User has “open a new map” or “loaded a previous saved map”.

basic path

•Using mouse, the User selects on the toolbar the component he wants to add.

•Using mouse, User selects the place where he wants to add the component on the Main Window. User indicates with mouse the two opposite corners of the component.

3       Cost Calculation

Three different methods were used to calculate the final cost of the product. The results of the different methods are compared at the end of this section.

 

3.1    Algorithmic Estimation

We come up with a Equation to calculate the cost of the project following COCOMO Strategy which is

Effort = a* (Size) b * c

Where.

a   = 2.4

b   = 1.05                  [a, b Are the parameters depending on the project type.]

Size = lines/1000 of code.

C     = 1.2                    [Adjustment factors: cost driver such as Required software

  Reliability, Database size]

             

Therefore, 2.4(1) 1.05 * 1.2 = 3.02 person/ month

= ~ 3 person/ months.

= 40Hours * 4 weeks * 3

= 480 Hours

 

Our total time is 3 month to complete and we are 5 people of us in our group to complete the project.

Therefore, it gives us,

                 = 8h/week for each person.

                 = 32h/month for each person.

 

For 5 person = 160 hours* 30 $/Month

                      = 4800 $/Month

 

With 2.125 times overhead it becomes (4800+ 10200)= 15000 $ per month for 5 people.

 

Total cost = 3 * 15000 $

                 = 45000 $ (With include overhead)
